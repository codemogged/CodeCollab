# Product Direction

## Positioning

CodeBuddy should position as:

The easiest place to build software with friends.

Not:
- another AI IDE
- another generic no-code app builder
- another project management board

## Ideal first users

### Segment 1: friend groups with one technical anchor
A small group wants to build something together, but only one person really knows how software projects usually work.

### Segment 2: ambitious non-technical founders
They want software progress, but they need the product to translate technical work into plain English.

### Segment 3: creators and communities
People with audiences or niche expertise want to turn ideas into tools without forming a full startup team first.

## MVP scope

### Must have
- project rooms
- shared prompting
- simple task board
- GitHub repo connection
- AI-generated project summaries in plain English
- activity feed based on issues, branches, pull requests, and deploys
- friend invites and lightweight permissions

### Should have
- AI-generated project plan from one initial idea
- simplified approval flow for code changes
- real-time comments and reactions
- role for vetted expert contributors

### Later
- payment rails for hiring experts
- templates by project type
- onboarding for kids and classrooms
- voice notes and multimodal project input

## Design principle

The main UX standard should be:
A smart child should be able to understand what the app is doing, what is blocked, and what happens next.

That means:
- no developer jargon on primary screens
- every repo action gets translated into plain language
- one primary call to action per view
- visible progress at all times

## Defensibility

Potential defensibility comes from:
- the simplified collaboration model
- the GitHub-backed execution graph
- trusted marketplace supply if expert help is added
- distribution through social and friend-group usage instead of just developer acquisition
