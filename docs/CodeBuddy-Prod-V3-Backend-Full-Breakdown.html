Section 00 — Architecture Overview

CodeBuddy is an Electron desktop application with a Next.js renderer. All backend logic executes in the Electron main process (full Node.js). The renderer communicates with the main process exclusively through IPC (Inter-Process Communication) via a preload bridge.

Service Creation Order (in main.js)

  1. processService — child process management
  2. settingsService — settings persistence
  3. toolingService — tool detection & AI providers (depends on processService)
  4. activityService — in-memory event log
  5. sharedStateService — .codebuddy/ directory management
  6. p2pService — peer-to-peer collaboration (depends on sharedStateService)
  7. fileWatcherService — file system watching (depends on repoService, processService, p2pService)
  8. projectService — project lifecycle (depends on settings, tooling, p2p, sharedState)
  9. repoService — Git operations (created async, may lag slightly)
Key Pattern: The sendEvent function is initially () => undefined in main.js. It gets wired up to the actual BrowserWindow's webContents.send only after IPC handlers are registered. All services that need to emit events to the renderer receive this function.

Section 02 — Main Process main.js

2.1 App Lifecycle

  • VM Detection & GPU Disable: Checks os.cpus() for VM CPU models (QEMU, Virtual, VMware, Xen). If detected, adds --disable-gpu via app.commandLine.appendSwitch.
  • Hardware Acceleration: Disabled via app.disableHardwareAcceleration().
  • Single Instance Lock: Not implemented (no requestSingleInstanceLock).

2.2 Diagnostic Logger

const LOG_FILE = path.join(app.getPath("userData"), "codebuddy-debug.log");

All console.log/warn/error calls are intercepted and tee'd to a file with ISO timestamps. Truncated to last 5000 lines on startup to prevent unbounded growth.

2.3 Static Server

ensureStaticServer()

In production, Next.js exports static HTML to out/. This function creates an HTTP server that serves those files on a random available port (starting at 3000). Returns the URL, e.g., http://localhost:3000. The server is stored in _staticServer and closed on quit.

2.4 Window Creation

createWindow()

  • Dimensions: 1400×900 default, 900×600 minimum
  • Title: CodeBuddy [build copilot-fix-v3]
  • Preload: electron/preload.js
  • Security: nodeIntegration: false, contextIsolation: true, webviewTag: true
  • Dev vs Prod: Dev loads http://localhost:3000; Prod loads from static server
  • Cache: Production clears clearCache + clearCodeCaches on start
  • Webview security: Guest pages get contextIsolation: true, sandbox: true, nodeIntegration: false
  • X-Frame-Options stripping: Strips X-Frame-Options headers from localhost URLs (enabling preview iframe)

2.5 Service Bootstrap

bootstrapDesktopServices(mainWindow)

Called from app.whenReady(). Creates all services in dependency order (see Section 00), then calls registerIpcHandlers() passing all services + sendEvent. Also creates repoService asynchronously.

2.6 Cleanup

cleanupBeforeQuit()

Called on before-quit and will-quit events. Stops file watcher, leaves all P2P rooms, closes static server.

Section 04 — IPC Handler Registry register-handlers.js

4.1 safeHandle Pattern

function safeHandle(channel, handler) {
  try { ipcMain.removeHandler(channel); } catch {}
  ipcMain.handle(channel, handler);
}

Removes any existing handler before registering. Prevents "handler already registered" crashes during hot reload.

4.2 P2P State Change Handler

This is the most complex handler in the file — registered via p2pService.onStateChange(). Handles 7 categories of incoming state changes from peers:

Category: plan

Forward-only merge: Uses STATUS_ORDER (planned:0 → building:1 → review:2 → done:3). When remote plan arrives, iterates all tasks. If the remote task status is MORE advanced than local, accepts it. If local is more advanced, preserves local. Also merges taskThreads (keeps version with more messages per thread). Uses settingsService.atomicUpdate() for race-free merge.

Category: tasks

Individual task status sync. Finds the project by matching data.projectId or the peer's joined project. Updates the single task's status using updateTaskStatusInPlan(). Deriving subproject status from child tasks.

Category: conversation

Appends peer messages to the correct conversation thread. Supports 3 types: project-manager (PM conversation), task-agent (task threads), solo-chat (solo sessions). Marks messages with fromPeer: true and peerName. De-duplicates by message ID.

Category: thread-sync

Full history merge triggered on peer connect. Merges all taskThreads, conversation, and soloSessions — keeps the version with more messages per thread. Atomic update.

Category: new-commits

Auto-pull on codebuddy-build branch. If fileWatcher reports agent is active, defers the pull. Otherwise calls fileWatcherService.autoPull().

Category: main-updated

Fetches origin/main and fast-forward merges if currently on main. If not on main, just fetches the ref.

Category: agent-context

Writes signal marker JSON to .codebuddy/agents/context/ so peer machines can load prior agent knowledge.

4.3 Peer-Ready Handler

Registered via p2pService.onPeerReady(). When a new peer connects, automatically broadcasts all local threads, PM conversation, and solo sessions for full sync.

4.4 Agent Lifecycle Guards

All agent-invoking handlers (project:sendTaskMessage, project:sendPMMessage, project:sendSoloMessage, project:generatePlan, project:generateTaskPrompt) wrap their logic in a try/finally that:

  1. Sets fileWatcherService.setAgentActive(true) before running
  2. Sets fileWatcherService.setAgentActive(false) after completion
  3. Triggers fileWatcherService.doAutoSync() to commit any agent-made file changes

4.5 syncInProgress Guard

A module-level boolean that prevents project:savePlan from overwriting plan.json during a project:syncWorkspace import.

4.6 Event Sender Wiring

After all handlers are registered, the function wires sendEvent into services that need it:

projectService.__setEventSender?.(sendEvent);
processService.__setEventSender(sendEvent);
activityService.__setEventSender(sendEvent);
p2pService.__setEventSender(sendEvent);

5.6 Atomic Write System

  1. All reads and writes are serialized through a Promise queue (enqueue(fn))
  2. _writeToDisk(data): write to .tmp → rename existing to .bak → rename .tmp to settings.json

  4. normalizeSettings(raw): merges raw with defaults, normalizes all projects
  5. normalizeProject(p): ensures every project has a valid dashboard with all required arrays

  2. Abort stuck rebase (force-remove .git/rebase-merge and .git/rebase-apply dirs if rebase --abort fails)

8.3 AI Provider Routing

runCopilotPrompt({prompt, cwd, allowTools, timeoutMs, model})

Runs gh copilot -p [prompt] with optional --model and --allow-tool flags.

runGenericPrompt({prompt, cwd, timeoutMs, model})

Provider-aware routing. Resolves to copilot/claude/codex based on feature flags + model ID:

  • Claude: claude -p [prompt] --dangerously-skip-permissions
  • Codex: codex exec --full-auto (prompt via stdin to avoid Windows EINVAL issues)
  • Copilot: gh copilot -p [prompt] --no-color -s

resolveProviderForPrompt()

Logic: single provider enabled → use it. Model ID in claude catalog → claude. Model ID in codex catalog → codex. Fallback priority: copilot → codex → claude.

9.3 Command Path Resolution

Refreshes PATH, reads settings for configured CLI paths, resolves binary locations for git, gh, copilot, claude, codex. Auto-installs copilot if not found and enabled. Returns a resolved command object used by all agent functions.

9.4 Safe Command Jail

getSafeCommandJailDir()

Creates a temporary directory with no-op wrapper scripts that shadow dangerous commands (code, explorer, start, taskkill, powershell, vim, etc.). This directory is prepended to PATH when running AI agents, preventing copilot/claude from launching external apps.

9.5 Agent Process Management

runProgram(file, args, cwd, requestMeta, stdinData)

The core function that runs AI CLI tools. Key behaviors:

  • Filters VS Code directories out of PATH
  • Prepends safe command jail directory
  • Overrides EDITOR, VISUAL, GIT_EDITOR, BROWSER env vars
  • Stores process as activeChildProcess (only one AI agent at a time)
  • Streams stdout/stderr via project:agentOutput events
  • Broadcasts tokens to P2P peers every chunk
  • 15-second keepalive interval for P2P peer stream
  • Accumulates last 12KB of output in activeRequestOutput
  • Handles .cmd files specially on Windows (shell: true)
  • Closes stdin immediately (or writes stdinData) to prevent CLI hangs

9.6 Project CRUD

createProject(payload)

Two modes:

  • Import existing directory: Takes importExistingPath. Initializes git if needed, creates GitHub remote, switches to codebuddy-build branch, creates project entry.
  • Create new directory: Creates folder, writes README.md + .gitignore, git init, initial commit, GitHub remote, push main, create codebuddy-build branch.

Both modes call syncSharedAgentContextFiles() and saveProject().

deleteProject(payload)

Options: deleteLocalFiles (rm -rf repoPath + empty parent), deleteGithubRepo (gh repo delete --yes). Updates settings atomically.

setActiveProject(projectId)

Updates activeProjectId in settings.

saveProject(project)

Saves project to settings (atomic), broadcasts config summary to P2P peers. Updates recentRepositories and workspaceRoots (max 8).

9.7 Plan Generation

generateProjectPlan(projectId, prompt, model)

  1. Gathers project snapshot (file tree depth 2, key files: package.json, README.md, etc.)
  2. Builds full prompt with system prompt + schema + project context + user request
  3. Runs CLI via runProviderCli() (no tool use, text-only)
  4. Parses JSON from output via parseJsonObjectFromText()
  5. Normalizes plan via normalizeGeneratedPlan()
  6. Saves project with updated dashboard

Section 10 — P2P Collaboration Service p2p-service.js

10.1 Technology Stack

  • Hyperswarm: DHT-based peer discovery (no central server)
  • Yjs CRDTs: Conflict-free replicated data types for state merging
  • Topic derivation: SHA-256 hash of codebuddy:{remoteUrl}

10.2 Session Architecture

Each project gets its own session (keyed by projectId) containing:

{ projectId, swarm, topic, repoPath, member, remoteUrl,
  ydoc, peers (Map), peerStreamAccumulators (Map),
  heartbeatInterval, reconnectTimer, reconnectAttempts,
  isJoined, lastPeerSeenAt }

10.3 Connection Lifecycle

  1. joinProject(): Creates session, initializes Yjs doc, loads shared state, connects swarm
  2. connectSwarm(): Creates Hyperswarm, joins topic (server + client), starts heartbeat (5s)
  3. handlePeerConnection(): Sets up per-peer state, sends "hello" with Yjs state vector
  4. handlePeerMessage(): Processes 6 message types (hello, yjs-update, yjs-sync, heartbeat, chat-token, chat-message, state-change)
  5. leaveProject(): Cleans up timers, destroys connections, Yjs doc, swarm

10.4 Message Types

TypePurpose
helloInitial handshake. Exchanges member profile + Yjs state vector for sync.
yjs-updateIncremental Yjs CRDT update (base64 encoded)
yjs-syncFull Yjs state sync response (base64 encoded)
heartbeatEvery 5s. Contains member info. Updates peer lastSeen.
chat-tokenReal-time AI token streaming. Accumulated per-peer (max 8KB). Includes scope, taskId, sessionId.
chat-messageComplete AI message. Clears token accumulator.
state-changeGeneric state broadcast. Triggers stateChangeCallbacks. Synced to Yjs + disk.

10.5 Reconnection

Exponential backoff: base 3s × 1.5^attempts, max 30s, max 10 attempts. Auto-triggers if no peers for 60 seconds.

10.6 Yjs State Maps

The Yjs document stores 4 maps: conversations (Array), members (Map), tasks (Map), agents (Map). Loaded from .codebuddy/ on join. Updates are broadcast to peers automatically via Yjs observers.

Section 11 — File Watcher Service file-watcher-service.js

11.1 Purpose

Watches the active project directory. After file changes stop for 10 seconds, auto-commits and pushes to codebuddy-build branch. Coordinates with agent activity and P2P pull operations.

11.2 Debounce & Guards

  • DEBOUNCE_MS: 10,000ms (10 seconds after last change)
  • paused: Set during pull operations to prevent echo loops
  • syncing: Set during auto-sync to prevent re-entry
  • agentActive: Set while AI agents are running. Defers auto-sync until agent finishes.

11.3 Ignored Directories

node_modules, .next, dist, dist-electron, .git, out, tmp,
__pycache__, .venv, venv, target, .cache, coverage, build

Also ignores .codebuddy/checkpoints/ and *.log files.

11.4 Auto-Sync Flow (doAutoSync())

  1. Clean up stuck git state (index.lock, rebase, merge)
  2. Ensure git identity is configured (tries gh API for username/email)
  3. Ensure on codebuddy-build branch (stash → switch → pop)
  4. Check for changes (git status --porcelain)
  5. Stage all (git add -A)
  6. Commit (git commit -m "auto: sync changes {timestamp}" --no-verify)
  7. Pull before push (git pull origin codebuddy-build --rebase)
  8. Push (git push origin codebuddy-build)
  9. Broadcast new-commits to P2P peers

Push recovery strategies (in order):

  1. If no upstream → push -u origin codebuddy-build
  2. If non-fast-forward → pull --rebase → push retry
  3. If rebase conflicts → fetch → soft-reset to remote → re-commit → push
  4. Last resort → push --force-with-lease

11.5 Push to Main (pushToMain(repoPath))

  1. Auto-sync pending changes first
  2. Switch to main
  3. Merge codebuddy-build → main (no-edit)
  4. Push main
  5. Switch back to codebuddy-build
  6. Broadcast main-updated to P2P peers

11.6 Auto-Pull (autoPull(repoPath))

Triggered by P2P new-commits signal. Pauses watcher, stashes changes, pulls --rebase, pops stash. Resumes watcher after 3s delay. Handles detached HEAD, non-codebuddy-build branches (fetch-only), and stash pop conflicts (drops stash, keeps remote).

11.7 Startup Housekeeping

  • Adds .codebuddy/checkpoints/ to .gitignore if missing
  • Configures git credential helper (gh auth git-credential)
  • Won't restart if already watching the same repo

Section 17 — Safety & Security Mechanisms

17.1 Electron Security

  • nodeIntegration: false — Renderer has no Node.js access
  • contextIsolation: true — Preload runs in isolated context
  • sandbox: true for webview guests
  • X-Frame-Options stripped only for localhost URLs
  • openExternal validates URL scheme (http/https only)

17.2 AI Agent Safety

  • Safe Command Jail: No-op wrappers shadow dangerous commands (code, explorer, taskkill, powershell, vim, etc.)
  • PATH filtering: VS Code directories removed from PATH during agent execution
  • Environment overrides: EDITOR=cat, VISUAL=cat, GIT_EDITOR=cat, BROWSER=echo
  • System prompt rules: Comprehensive safety instructions forbidding port-killing, external app launches, long-running processes
  • Only one active agent: Single activeChildProcess at a time
  • Checkpoint before every message: Full file snapshot before any agent interaction

17.3 Git Safety

  • GIT_TERMINAL_PROMPT=0 — Prevents interactive credential prompts
  • GCM_INTERACTIVE=never — Git Credential Manager non-interactive
  • cleanupGitState() — Automatic recovery from stuck index.lock, rebase, merge
  • --no-verify on auto-commits — Prevents pre-commit hooks from blocking sync
  • --force-with-lease as last resort (not --force)
  • Branch guard: ensureOnCodebuddyBuild() before agent file modifications

17.4 Data Safety

  • Atomic writes: Settings use tmp → backup → rename pattern
  • Targeted merge: Task agent results merge only changed fields into fresh settings, avoiding stale overwrites
  • Forward-only status merge: P2P never regresses task status (planned→building→review→done)
  • syncInProgress guard: Prevents savePlan from overwriting during syncWorkspace
  • Message deduplication: P2P conversation sync checks message IDs

Section 18 — Known Gotchas & Recovery Patterns

Git Index Lock

Symptom: "fatal: Unable to create '.../.git/index.lock': File exists"

Cause: Previous git operation was interrupted (process kill, crash).

Recovery: cleanupGitState() automatically deletes .git/index.lock. Called before every git operation in file-watcher and repo-service.

Detached HEAD

Symptom: git rev-parse --abbrev-ref HEAD returns "HEAD".

Cause: Aborted rebase left HEAD detached.

Recovery: inspectRepository() auto-checks out codebuddy-build. autoPull() also handles this.

Port 3000 Conflict

Symptom: AI agent tries to kill port 3000 or start a dev server on it.

Prevention: System prompt explicitly forbids killing port 3000 (CodeBuddy itself). Safe command jail blocks taskkill/kill. launchDevServer() detects conflicts.

Stale Settings Overwrite

Symptom: P2P-synced data lost after agent completes.

Prevention: sendTaskMessage uses targeted merge — reads FRESH settings inside atomicUpdate() and merges only its specific thread + status change. Does not overwrite the entire project.

Windows EINVAL with .cmd Files

Symptom: "EINVAL" error when spawning .cmd files with long arguments.

Cause: Node.js CVE-2024-27980 security fix rejects .cmd with suspicious characters.

Recovery: runProgram() detects .cmd files and uses shell: true mode. Codex CLI passes prompts via stdin instead of args.

Auto-Sync Echo Loop

Symptom: Infinite push/pull cycle between machines.

Prevention: File watcher pauses during pulls (3s resume delay). agentActive flag defers sync during agent runs. savePlan skips git push when P2P is live or agent is active.

Codex ChatGPT Model Error

Symptom: "not supported when using Codex with a ChatGPT account"

Recovery: runProviderCli() auto-retries without --model flag, letting the server pick its default.

No Git Identity

Symptom: "Author identity unknown" on commit.

Recovery: Both ensureGitIdentitySync() (file-watcher) and ensureGitIdentity() (project-service) detect missing identity and set it from gh api user or fallback to "CodeBuddy <codebuddy@local.invalid>".

Handler Already Registered

Symptom: Crash on hot reload — "Attempted to register a second handler for..."

Prevention: safeHandle() removes existing handler before registering new one.
