1. What Is CodeBuddy?

1.1 Elevator Pitch

CodeBuddy is a desktop application that lets anyone — even non-programmers — describe a software idea in plain English and have AI agents build it for them, step by step, all within one self-contained window. Think of it as a project manager, code editor, terminal, and live preview all rolled into one app, with AI doing the heavy lifting.

1.2 Who It's For

  • Non-technical creators who have app ideas but can't code.
  • Beginners who want to learn by watching an AI agent work.
  • Solo developers who want AI assistance with planning and routine tasks.
  • Friend groups who want to collaborate on side projects without complex tooling.

2. Architecture Overview

2.1 Three-Layer Sandwich

CodeBuddy's architecture is a three-layer sandwich. Each layer has a specific job and can only talk to the one next to it:

┌─────────────────────────────────────────────────────────────┐ │ RENDERER (Frontend) │ │ Next.js pages • React components • Tailwind CSS │ │ Runs in an Electron BrowserWindow (like a browser tab) │ ├─────────────────────────────────────────────────────────────┤ │ PRELOAD BRIDGE │ │ electron/preload.js → exposes window.electronAPI │ │ Secure channel: renderer can ONLY call approved methods │ ├─────────────────────────────────────────────────────────────┤ │ MAIN PROCESS (Backend) │ │ electron/main.js + 6 services │ │ Node.js with full system access: filesystem, processes, │ │ git, GitHub CLI, Copilot CLI │ └─────────────────────────────────────────────────────────────┘

Why this matters: The frontend (what you see) cannot directly access your filesystem or run programs. It must ask the backend through the preload bridge. This keeps things secure — a web page can't randomly delete your files.

2.2 How the Layers Talk (IPC)

IPC stands for Inter-Process Communication. It's how the frontend sends requests to the backend and gets results back. Think of it like a restaurant:

  • Frontend = the customer placing an order
  • Preload bridge = the waiter who takes the order and brings back food
  • Backend services = the kitchen that actually cooks

2.3 The AI Brain (Copilot CLI)

Every single AI interaction in CodeBuddy goes through one executable: copilot.exe (GitHub Copilot CLI). This is a command-line program that takes a text prompt and returns an AI-generated response.

Here's the basic flow:

User types → Frontend sends IPC → Backend builds prompt → copilot.exe runs → Response parsed → UI updated Detailed: 1. User clicks a button or types a message 2. Frontend calls window.electronAPI.project.someMethod(...) 3. IPC handler in register-handlers.js routes to project-service.js 4. project-service.js: a. Reads current project state from settings.json b. Gathers context (file trees, config files, conversation history) c. Builds a detailed text prompt with system instructions d. Calls processService.runProgram(copilotExePath, ["-p", prompt, ...flags]) 5. copilot.exe talks to the AI model (GPT-5.2, Claude, etc.) and returns text 6. project-service.js parses the response, updates settings.json 7. Response flows back through IPC to the frontend 8. React updates the UI
Key insight: CodeBuddy doesn't contain its own AI model. It relies entirely on GitHub Copilot CLI, which means the user needs a GitHub Copilot subscription. The CLI handles authentication, model selection, and rate limiting automatically.

2.4 Where Data Lives

CodeBuddy stores all of its data in two places:

LocationWhat's StoredExample Path
Settings file All project metadata, conversation history, plans, task threads, preferences %APPDATA%\codebuddy\settings.json
Project folders Actual source code, .codebuddy/ subdirectory with checkpoints and agent context files C:\Users\you\Projects\my-app\

The .codebuddy/ folder inside each project contains:

  • .codebuddy/checkpoints/ — Snapshots of all project files taken before every AI prompt (so you can undo)
  • .codebuddy/agents/ — Markdown files that give each task agent its full context

4. Backend Services Deep Dive

4.1 Project Service — The Brain (~2,660 lines)

File: electron/services/project-service.js

This is the largest and most important file in the entire codebase. It handles:

  • Creating and deleting projects (new or imported)
  • All 6 agentic AI calls (plan generation, PM chat, task chat, auto-prompt, dev server launch, GitHub repo)
  • The checkpoint/restore system
  • Agent context file synchronization
  • Safe command jail for protecting the app from AI mistakes
  • Git identity resolution and initial commit handling
  • README auto-generation from plans
Generalizability note: This service is framework-agnostic. It works by reading whatever config files exist in the project directory (package.json, Cargo.toml, pyproject.toml, go.mod, etc.) and passing them to the AI. It doesn't have hardcoded assumptions about what language or framework the user chose.

4.2 Process Service (~150 lines)

File: electron/services/process-service.js

Spawns and manages child processes (shell commands, dev servers, copilot.exe). Two main methods:

  • run(command, cwd, env) — Runs a shell command (e.g., npm install). Uses shell: true.
  • runProgram(file, args, cwd, env) — Runs a specific executable directly (e.g., copilot.exe). Uses shell: false.

Tracks all running processes in a Map. Emits events: process:started, process:output, process:completed, process:error, process:cancelled, process:timeout.

4.3 Repo Service (~200 lines)

File: electron/services/repo-service.js

Wraps Git operations using execFile(git, args, {cwd}). Provides:

  • inspectRepository() — Gets current branch, branches list, changed files, recent commits
  • listDirectory() / readFileContent() / writeFileContent() — CRUD for project files
  • stageFiles() / unstageFiles() / commit() — Git staging and committing
  • checkoutBranch() / getCommitDetails() / getFileDiff() — Branch management and diffs

4.4 Settings Service (~200 lines)

File: electron/services/settings-service.js

Reads and writes the central settings.json file. Also defines the two main system prompts (see Section 5). Key functions:

  • readSettings() — Reads and normalizes settings with defaults
  • updateSettings(partial) — Deep-merges partial updates
  • Broadcasts settings:changed events to the frontend whenever settings update

4.5 Tooling Service (~150 lines)

File: electron/services/tooling-service.js

Checks which CLI tools are available on the user's system. Detects: git, gh (GitHub CLI), copilot.exe, node, npm, claude (Code). Returns version info and availability status for each.

4.6 Activity Service

File: electron/services/activity-service.js

Logs timestamped activity events. Every IPC handler in register-handlers.js automatically logs events through this service. Events are stored in the project dashboard's activity array.

5. Deep Dive: Every Agentic Process

CodeBuddy has 6 distinct AI agent calls. Each one sends a carefully constructed prompt to Copilot CLI and handles the response differently. Below is every single one, with its full system prompt, context window contents, and how the response is used.

5.1 Generate Project Plan

Function: generateProjectPlan(projectId, prompt, model)

When it fires: When the user clicks "Generate Plan" on a new project, or when an imported project needs analysis.

Copilot CLI flags: -p <prompt> --no-color -s [--model <model>] (text-only, no tool use)

What goes into the prompt (the "context window"):

  1. System prompt — Either DEFAULT_SYSTEM_PROMPT_MARKDOWN (for new projects) or IMPORTED_PROJECT_SYSTEM_PROMPT (for imported projects). These are ~40-80 line markdown documents that tell the AI its role and rules.
  2. JSON schema — The exact JSON structure the AI must return (subprojects, tasks, build order, etc.)
  3. Project name and description
  4. File tree — A depth-2 listing of all files in the project (from gatherProjectSnapshot())
  5. Key file contents — The actual contents of important files (package.json, README, etc.)
  6. User's prompt — The description they typed when creating the project

System Prompt for New Projects (condensed):

# CodeBuddy Project Planner You are the project planning system for CodeBuddy. Your job is to turn a non-technical user's product request into a practical MVP plan. Goals: - Make the plan understandable to someone with no coding experience - Break the MVP into clear subprojects (2-5) - Create concrete implementation tasks (2-5 per subproject) in build order - Keep the first version narrow, testable, and realistic CodeBuddy Platform Context: - Built-in Terminal tab for running shell commands - Built-in Live Preview tab for viewing web apps - Built-in file editor and Git integration - NEVER reference VS Code, external terminals, or browsers - Starting prompts must reference CodeBuddy's native tools Output: Return valid JSON matching the requested schema exactly.

System Prompt for Imported Projects (condensed):

# CodeBuddy Project Analyst Your job is to examine an already-built codebase and produce a clear picture of what exists, what works, and what the developer should focus on next. Analysis rules: - Set task status to "done" if code clearly exists, "building" if partial, "planned" if needed - Put completed subprojects first, in-progress next, then planned - The dashboard should feel like a progress report, not a blank plan

What happens with the response:

  1. Parse the JSON response into subprojects, tasks, and build order
  2. Generate unique IDs for each subproject and task
  3. Assign due dates (spread across ~14 days from now)
  4. Store everything in project.dashboard.plan
  5. Write agent context files to .codebuddy/agents/
  6. Auto-generate a README.md with the plan summary
  7. Add a "plan generated" activity event
  8. Set project stage to "Building"

5.2 PM Chat (Project Manager Conversation)

Function: sendPMMessage({ projectId, prompt, model, attachedFiles, replaceFromMessageId })

When it fires: Every time the user sends a message in PM Chat mode.

Copilot CLI flags: -p <prompt> --no-color -s [--model] (text-only, no tool use)

What goes into the prompt:

  1. System prompt markdown — The project's stored system prompt (usually the planner prompt)
  2. Role instructions: "You are the Project Manager. The plan has already been created. Do NOT regenerate or modify the plan."
  3. CodeBuddy platform rules — Never reference VS Code, external terminals, etc.
  4. "Attention User Input Required" instructions — If the response needs user-provided info (API keys, etc.), it must end with a clearly marked section
  5. Project name and description
  6. Recent conversation transcript — Previous messages in the chat
  7. Attached files — Any files the user drag-dropped into the message
  8. The user's latest message

What happens before sending:

  • A checkpoint snapshot is created — all project files are copied to .codebuddy/checkpoints/<id>/
  • This allows edit/retry to restore to the exact state before this message

What happens with the response:

  • The raw text is stored as an AI message in dashboard.conversation
  • The user's message is also stored (with their name, initials, model used, checkpoint ID)
  • Agent context files are re-synced

5.3 Task Agent Chat (The Builder) — Most Powerful

Function: sendTaskMessage({ projectId, taskId, threadId, prompt, model, attachedFiles, replaceFromMessageId })

When it fires: Every time the user sends a message while on a task (URL has ?task=TASK_ID).

Copilot CLI flags: -p <prompt> --allow-all-tools --add-dir . --no-color -s [--model]

This is the only agent that can modify files. The --allow-all-tools flag gives Copilot CLI permission to read and write files, run commands, and use other tools. The --add-dir . flag adds the project directory as context. All other agents are text-only.

What goes into the prompt (in order):

  1. Task agent system prompt — Built by buildTaskAgentSystemPrompt(). Includes:
    • "You are a hands-on task agent inside CodeBuddy"
    • Response format: ## What I did, ## Recommended next steps, ## Move to next task when
    • Full CodeBuddy platform rules (26 lines of restrictions)
    • README.md reading/updating instructions
    • "Attention User Input Required" instructions
    • TASK_STATUS metadata line instructions
    • Task title, note, and subproject context
  2. CRITICAL SAFETY RULES — A 30-line block of absolutely forbidden commands (see Section 8)
  3. Context markdown — The shared task context from .codebuddy/agents/
  4. Attached files
  5. Recent conversation transcript
  6. The user's latest message

The Task Agent System Prompt (full):

You are a hands-on task agent inside CodeBuddy — a desktop coding workspace. Do the task work and reply like a collaborator in chat. Prefer short sections, bullets, and concrete next steps over schemas. Assume the user is non-technical and avoid unnecessary jargon. Default response structure: ## What I did, ## Recommended next steps, ## Move to the next task when. CRITICAL — CodeBuddy Platform Rules: CodeBuddy RUNS ON PORT 3000. NEVER kill, stop, or interfere with port 3000. - Built-in TERMINAL panel for shell commands - Built-in LIVE PREVIEW panel for localhost - NEVER tell the user to open VS Code, an external terminal, or browser - NEVER run 'code .', 'explorer', 'open', or launch external applications - NEVER suggest the user leave CodeBuddy - NEVER kill processes or run taskkill/Stop-Process/kill/pkill - If port 3000 is in use, that IS CodeBuddy — use a different port README.md: Read it before starting. Update it after completing work. User Input Required: If response needs API keys, credentials, etc., end with: --- ## Attention User Input Required [list items needed] End EVERY reply with: TASK_STATUS: planned|building|review|done Optionally: TASK_STATUS_REASON: <short reason> Task title: [from plan] Task note: [from plan] Subproject: [from plan]

What happens with the response:

  1. Extract TASK_STATUS metadata from the last lines
  2. Update the task's status in the plan (e.g., "planned" → "building")
  3. Store messages in the task's thread
  4. Re-sync agent context files
  5. Format the response (if it accidentally returned JSON, convert to readable markdown)

5.4 Auto-Prompt / Generate Task Prompt

Function: generateTaskPrompt({ projectId, taskId, threadId, model })

When it fires: When the user clicks the ⚡ (lightning) auto-prompt button.

Copilot CLI flags: -p <prompt> --no-color -s [--model] (text-only)

What goes into the prompt:

  1. "You generate the next best user prompt for a CodeBuddy task"
  2. Instruction to return JSON: {"prompt":"string","taskStatus":"planned|building|review|done","reason":"string"}
  3. CodeBuddy platform context rules
  4. Project name and description
  5. Subproject and task details (title, note, current status, starting prompt)
  6. Latest thread summary and recent conversation transcript (last 10 messages)

What happens with the response:

  • Parse the JSON to get the generated prompt text, task status assessment, and reason
  • Update task status if the AI says it's done
  • Return the prompt text to the frontend, which pre-fills the chat composer
  • If auto-advance is enabled and the task is "done", the frontend auto-navigates to the next incomplete task

5.5 Launch Dev Server

Function: launchDevServer({ projectId, model })

When it fires: When the user clicks "Run App" to preview their project.

Copilot CLI flags: -p <prompt> --no-color -s [--model] (text-only, fast)

What goes into the prompt:

  1. "You are a dev-ops expert"
  2. Rules: output EXACTLY two lines: LAUNCH_COMMAND: and EXPECTED_PORT:
  3. Platform info (Windows/macOS/Linux, use npm.cmd on Windows)
  4. Port detection hints (check package.json scripts for hardcoded PORT, .env files, vite.config, framework defaults)
  5. Safety: never launch Electron, Tauri, or desktop shells; only start the web renderer
  6. Project name, file tree, all config file contents (up to 2KB each), README excerpt

What happens with the response:

  1. Parse LAUNCH_COMMAND: and EXPECTED_PORT: from the response
  2. If the agent fails, fall back to a heuristic detectLaunchCommand() that checks package.json scripts
  3. Return both values to the frontend for spawning the dev server

Special feature: .env auto-copy

Before asking the AI, the function checks if .env exists. If not, it copies from .env.example, .env.local.example, .env.sample, or .env.template. This prevents first-run failures.

5.6 Ensure GitHub Repo

Function: ensureGithubRepoForProject(projectId)

When it fires: When the user clicks "Push to GitHub" or during project creation with GitHub enabled.

No AI involved — this is a pure Git/GitHub CLI operation.

Steps:

  1. Resolve git identity (from gh api user or fallback to "CodeBuddy")
  2. Set user.name and user.email in git config
  3. Ensure an initial commit exists (stage all files + commit if needed)
  4. Check if remote "origin" already exists
  5. If not, run: gh repo create <name> --private --source . --remote origin
  6. Push to origin with: git push -u origin HEAD
  7. Save the GitHub URL in the project settings

6. Deep Dive: The "Run App" Button

This is one of the most complex flows in CodeBuddy, spanning the backend and frontend. Here's every step from click to preview:

User clicks "▶ Run App" │ ▼ ┌─── handleRunApp() ──────────────────────────────────────────────┐ │ 1. Call electronAPI.project.launchDevServer(projectId) │ │ → Backend reads all config files (package.json, etc.) │ │ → Backend asks AI: "What command starts this project?" │ │ → AI returns: LAUNCH_COMMAND + EXPECTED_PORT │ │ → Fallback: detectLaunchCommand() heuristic if AI fails │ │ 2. Why no PORT override? Earlier versions tried to inject a random port (like PORT=5847) into the environment, but many projects (e.g., Create React App with set PORT=3001 in their scripts) override env vars. The current approach lets the project decide its own port and uses AI prediction + multi-port probing to find it.

7. Deep Dive: The Autoflow System

The "autoflow" is not a single button — it's a system of three connected features that together create a semi-autonomous build loop:

7.1 Kickoff Prompts

When the plan is generated, every task gets a startingPrompt — a pre-written first message designed to tell the task agent exactly what to do. When you navigate to a task for the first time, a card appears:

┌──────────────────────────────────────────────────┐ │ ⚡ Kickoff prompt │ │ │ │ "Initialize the React project with Create React │ │ App. Set up the folder structure with │ │ components/, pages/, and utils/ directories. │ │ Install React Router and Tailwind CSS." │ │ │ │ [ Use kickoff ] │ └──────────────────────────────────────────────────┘

Clicking "Use kickoff" pastes this prompt into the chat composer and sends it.

7.2 Auto-Prompt (⚡ Button)

After the first message, the ⚡ button in the chat composer calls generateTaskPrompt(). This asks the AI: "Given where this task is right now, what should the user say next?" The AI returns a tailored prompt that gets pre-filled into the composer.

7.3 Auto-Advance

A toggle: "Auto-advance tasks." When enabled:

  1. After each AI response, the system checks if the task status is "done"
  2. If done, it finds the next incomplete task in the plan
  3. Automatically navigates to that task's chat
  4. The kickoff prompt appears for the new task

The Full Autoflow Loop

┌─ Start ─────────────────────────────────────────────────────────┐ │ │ │ Task 1: "Set up project structure" │ │ ├── Kickoff prompt appears → User clicks "Use kickoff" │ │ ├── Agent works, writes files, replies with TASK_STATUS: done │ │ ├── Auto-advance enabled → navigates to Task 2 │ │ │ │ │ Task 2: "Build the homepage" │ │ ├── Kickoff prompt appears → User clicks "Use kickoff" │ │ ├── Agent works, replies with TASK_STATUS: building │ │ ├── User clicks ⚡ → AI generates next prompt → pre-filled │ │ ├── User clicks Send → Agent continues → TASK_STATUS: done │ │ ├── Auto-advance → Task 3 │ │ │ │ │ ... continues through all tasks ... │ │ │ │ Last task: TASK_STATUS: done → All tasks complete! 🎉 │ └──────────────────────────────────────────────────────────────────┘
Current state: The autoflow requires the user to click "Use kickoff" or "Send" for each step. It's semi-automated, not fully autonomous. A fully autonomous mode (where it sends kickoff → waits → sends auto-prompt → waits → repeat) could be built on top of this foundation but doesn't exist yet.

8. Safety & Security Model

Since CodeBuddy gives an AI agent the power to run commands and write files, safety is critical. Here are the four layers of protection:

8.1 System Prompt Rules

Every agent prompt includes explicit rules about what the AI must never do. These are "soft" guardrails — they rely on the AI following instructions.

8.2 CRITICAL SAFETY RULES Block

The task agent (the only one with tool access) gets an additional 30-line safety block injected into every prompt:

=== CRITICAL SAFETY RULES — VIOLATION WILL CRASH THE APP === You are running INSIDE the CodeBuddy desktop application. CodeBuddy runs on port 3000 (Next.js dev server) and an Electron process. If you kill or interfere with port 3000, YOU WILL CRASH THE APP. ABSOLUTELY FORBIDDEN COMMANDS: - taskkill, Stop-Process, kill, pkill, killall - netstat + kill/stop combos - npx kill-port, fkill, lsof + kill - code, code ., code <file> - explorer, open, start, xdg-open - Any command that opens a GUI window, browser, or external terminal - npm start on port 3000 (it's taken by CodeBuddy) - shutdown, restart, logoff, exit ALLOWED COMMANDS: - npm install, pip install, cargo build (installing/building) - mkdir, touch, echo (creating files) - cat, ls, dir (reading files) - git add, commit, status (version control) - npm test, pytest, cargo test (testing) - Any command that reads, creates, or modifies PROJECT files === END SAFETY RULES ===

8.3 Safe Command Jail

A "hard" guardrail: Before running copilot.exe, CodeBuddy creates a temporary directory filled with no-op wrapper scripts for dangerous commands:

Jail directory contains fake executables for:
  code, code.cmd         → "exit 0" (VS Code)
  explorer, explorer.cmd → "exit 0" (Windows Explorer)
  open                   → "exit 0" (macOS open)
  start, start.cmd       → "exit 0" (Windows start)
  taskkill, taskkill.cmd → "exit 0" (process killer)
  shutdown               → "exit 0"
  tasklist               → "exit 0"

This jail directory is prepended to the PATH environment variable when running copilot.exe. So even if the AI tries to run code ., it executes the fake no-op version instead of the real VS Code.

8.4 Environment Filtering

When running copilot.exe, the environment is sanitized:

  • EDITOR=cat — Prevents any editor from opening
  • BROWSER=echo — Prevents browser launches
  • PATH is filtered to remove known dangerous tool directories
  • The safe command jail is prepended to PATH

9. Generalizability Analysis

The central question: Can CodeBuddy handle any type of project someone wants to build, not just React web apps?

9.1 What Makes It Generalizable (Strengths)

AspectHow It's GeneralizedEvidence
Config file detection Reads 15+ config formats launchDevServer() checks: package.json, Cargo.toml, pyproject.toml, requirements.txt, Gemfile, go.mod, build.gradle, pom.xml, Makefile, docker-compose.yml, composer.json, mix.exs, .csproj, .sln, index.html
AI-driven planning System prompts are framework-agnostic The planner prompt says "MVP plan" without mentioning specific frameworks. It defers to the AI's training data.
AI-driven dev server AI determines the launch command from context Instead of hardcoded npm start, the agent reads config files and outputs the right command (could be cargo run, python manage.py runserver, etc.)
Task agent tools Copilot CLI --allow-all-tools The task agent can read/write any file type, run any command — not limited to JS/TS
File tree scanning Uses generic gatherProjectSnapshot() Walks the directory, excludes common build artifacts (node_modules, .git, __pycache__, target/, etc.), reads key files
Model selection 15+ AI models including reasoning models Users can pick the best model for their project type (e.g., Claude Opus for complex architecture)

9.2 Where Generalizability Could Break (Gaps)

GapImpactMitigation Path
Preview is web-only The <webview> can only show HTTP servers. CLI tools, desktop apps, mobile apps won't have a preview. Could add terminal-only preview mode for non-web projects, or screenshot capture for desktop apps.
Port detection assumes HTTP Projects that don't start an HTTP server (game engines, CLI tools, desktop apps) won't benefit from "Run App" Could detect project type and skip preview for non-web projects, showing terminal output instead.
Windows-centric commands npm.cmd/npx.cmd hints are Windows-specific in the launch prompt Already handled: the prompt checks process.platform and adjusts. Cross-platform support exists.
Friends/collaboration is mock No real multi-user backend exists Need authentication, WebSocket for real-time sync, and a server component.
System prompt mentions "MVP" Could bias the AI toward simple web apps The prompt says "practical MVP plan" which is generally fine — the AI still adapts to the description.
Checkpoint storage Full file copies before every prompt can get large for big projects Could switch to git-based checkpoints (stash/tag) instead of file copies.

9.3 Verdict

CodeBuddy is well-generalized for any project that involves code files and can be developed from a terminal. It reads whatever project structure exists, lets the AI figure out the specifics, and doesn't hardcode assumptions about languages or frameworks. The main limitation is the preview system, which currently only works for web apps that start an HTTP server. Non-web projects (CLI tools, desktop apps, games, data science) can still use the planning, task agent, file editing, and terminal features — they just won't get the live preview.
